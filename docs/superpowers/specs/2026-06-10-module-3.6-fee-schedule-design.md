# Module 3.6 — Practice Fee Schedule: Design

**Status:** approved, ready for implementation plan
**Date:** 2026-06-10
**Depends on:** Module 3.5 (`cdt_codes` table). Independent of eligibility.
**Feeds:** Module 6 (co-pay calculation consumes per-practice fees).

## Goal

Let each practice set its own fee per CDT code in Settings, instead of typing the
fee on every procedure. Selecting a CDT code in the appointment/schedule view then
auto-fills the practice's fee. This turns `cdt_codes.default_fee_cents` (currently
global and null for all seeds) into a real per-practice value via an override table.

## Scope decisions (locked during brainstorm)

- **Catalog stays at the ~20-code seed.** 3.6 manages fees only over the existing
  seeded CDT codes. Full ADA D-code catalog seeding is deferred to Module 6.2. No
  CSV import and no server-side pagination in this module — at 20 codes a client-side
  filter is enough.
- **Endpoints are NOT nested under `/practices/{id}`.** Practice scope comes from
  auth context (`request.state.user.practice_id`), like every other router. This
  corrects the build-order sketch.
- **Path is keyed by the CDT code string** (`/fee-schedule/D0120`), not the UUID —
  human-readable, stable, and the frontend already holds the code. The table stores
  the `cdt_code_id` FK internally for integrity; the handler resolves code → id.
- **Money is absolute integer cents** everywhere, per the Phase 3 money convention.

## Data model

New table `practice_fee_schedule`:

| column        | type                          | notes                                   |
|---------------|-------------------------------|-----------------------------------------|
| `id`          | UUID PK                       |                                         |
| `practice_id` | UUID FK → practices, NOT NULL |                                         |
| `cdt_code_id` | UUID FK → cdt_codes, NOT NULL |                                         |
| `fee_cents`   | Integer, NOT NULL             | absolute cents                          |
| `created_at`  | timestamp (`TimestampMixin`)  |                                         |
| `updated_at`  | timestamp (`TimestampMixin`)  |                                         |
| `deleted_at`  | timestamp, nullable           | soft delete (revert = soft-delete row)  |

- **Unique constraint** on `(practice_id, cdt_code_id)` — one fee per code per practice.
  Scoped to non-deleted rows (partial unique index where `deleted_at IS NULL`) so a
  reverted code can be set again.
- Indexes on `practice_id` and `cdt_code_id`.
- `cdt_codes.default_fee_cents` remains the platform-wide fallback; the practice row
  overrides it.

### Fee resolution order

Used everywhere a fee is read (typeahead, fee-schedule list, future treatment-plan items):

```
practice fee (practice_fee_schedule.fee_cents, deleted_at IS NULL)
  → cdt_codes.default_fee_cents
  → blank (staff types it)
```

## API

Header-based practice scope, write-role guard, audit log line, and the idempotency
middleware — identical to the `insurance_plans` router.

### `GET /api/v1/fee-schedule`

Returns **every** CDT code joined with the practice's override, so the UI renders the
full catalog with current-or-blank fees in one call.

Row shape:
```
{
  cdtCodeId: UUID,
  code: string,            // "D0120"
  description: string,
  category: string,
  defaultFeeCents: int | null,   // global fallback
  practiceFeeCents: int | null,  // this practice's override, null if none
  resolvedFeeCents: int | null   // practiceFeeCents ?? defaultFeeCents
}
```

### `PUT /api/v1/fee-schedule/{code}`

Upsert this practice's fee for one code. Body `{ feeCents: int }` (must be ≥ 0).
Idempotency-Key required (middleware-enforced). Resolves `code` → `cdt_code_id`;
404 if the code is unknown/inactive. Upserts the override row (un-soft-deletes if a
reverted row exists). Returns the updated fee-schedule row (same shape as GET row).

### `DELETE /api/v1/fee-schedule/{code}`

Revert to default: soft-delete the practice's override row. Idempotent — deleting a
code with no override is a no-op 204. Returns 204.

All three require practice scope (403 if missing) and write-role
(`{admin, provider, front_desk}`, 403 otherwise). Each mutation emits an audit log
line with `practice_id` and `code`.

## Fee prefill in the appointment/schedule view

The existing `/cdt-codes` typeahead (used by `ProceduresSection`) becomes
**practice-fee-aware**: its response gains `resolvedFeeCents` (practice override if
set, else global default), computed via a join on `practice_fee_schedule` for the
request's practice.

`ProceduresSection.selectCdtCode` keeps its existing conditional prefill — it only
fills the fee when staff haven't typed one — now reading `resolvedFeeCents`:

```ts
fee:
  prev.fee || code.resolvedFeeCents == null
    ? prev.fee                                       // keep typed / leave blank
    : (code.resolvedFeeCents / 100).toString();      // autofill practice fee
```

Resulting behavior in the appointment view:
- Pick a code with a practice fee set → fee field autofills that fee.
- No practice fee but a global default exists → fills the global default.
- Neither → field stays blank (today's behavior).
- Staff already typed a fee → autofill does **not** clobber it.

The same typeahead + prefill hook is reused for treatment-plan item entry in a later
module — no reshaping needed.

## Frontend

New **"Fee Schedule"** tab in Settings, following the `AppointmentTypesSettings`
pattern (TanStack React Query + table):

- File: `apps/web/components/settings/FeeScheduleSettings.tsx`.
- Add `{ key: "fee-schedule", label: "Fee Schedule" }` to the Settings `TABS` array
  and render the component for that tab.
- Table of all CDT codes (from `GET /api/v1/fee-schedule`) with an editable dollar
  field per row. Save → `PUT` (dollars → cents). Clear the field → `DELETE` (revert
  to default); the row then shows the default as a placeholder.
- Small summary: "X of 20 customized".
- Client-side filter box over code/description (no server pagination at 20 codes).
- API client functions in a new/extended module under `apps/web/lib/api/`
  (`getFeeSchedule`, `setFee`, `revertFee`) with React Query hooks, mirroring the
  existing scheduling/procedures API modules.

- `create_table` for `practice_fee_schedule` with columns above.

## Testing

Shipped in the same change (happy path + failure case per repo convention).

**API integration** (`tests/integration/`):
- `GET /fee-schedule` returns all seeded codes, each with `resolvedFeeCents`
  reflecting override-or-default.
- `PUT /fee-schedule/{code}` sets a fee; GET then shows it as `practiceFeeCents` and
  `resolvedFeeCents`. Second identical PUT with same Idempotency-Key is a no-op.
- `PUT` with an unknown code → 404.
- `DELETE /fee-schedule/{code}` reverts: GET shows `practiceFeeCents` null,
  `resolvedFeeCents` falls back to default. DELETE on a non-overridden code → 204.
- **Practice isolation:** practice A's override is invisible to practice B's GET.
- **Write-role:** a read-only role gets 403 on PUT/DELETE.
- **Appointment-view autofill (typeahead):** with a practice fee set, the
  `/cdt-codes` search response for that practice returns `resolvedFeeCents` equal to
  the override; with no override, it equals the global default.

**Frontend** (`__tests__/`):
- Dollars ↔ cents conversion on save/display.
- `ProceduresSection` prefill: selecting a code with a `resolvedFeeCents` populates
  the fee field; selecting one with `null` leaves it blank; a pre-typed fee is not
  clobbered.

## Out of scope (this module)

- CSV bulk import of an existing fee schedule.
- Co-pay / patient-responsibility calculation (→ Module 6).
- Treatment-plan item fee prefill wiring (later module; reuses the same hook).
