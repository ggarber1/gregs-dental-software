# Module 5.2–5.4 — Eligibility Verification (sync slice)

**Status:** Design approved 2026-06-11
**Depends on:** 5.1 (insurance_plan / patient_insurance CRUD — done)
**Build order ref:** `docs/superpowers/specs/phase3-build-order.md` (row 3)
**Research source of truth:** `research/13_module5_insurance_verification.md`,
`research/17_clearinghouse_comparison.md`

This spec narrows the Module 5 research into a buildable, sync-first slice. It
covers the eligibility provider abstraction, the Stedi adapter, the
`eligibility_checks` store, a synchronous "verify now" API, and a patient-chart
benefit-summary card. The async batch pipeline is deliberately deferred to a
follow-up (see §9).

---

## 1. Scope

### In this spec
- `EligibilityProvider` interface + **Stedi adapter only**.
- `eligibility_checks` table (SQLAlchemy model + Alembic migration).
- **Synchronous** `POST /api/v1/eligibility/check` (calls Stedi inline, 30s
  timeout) + two retrieval endpoints.
- Frontend: **benefit-summary card** on the patient chart + **"Verify now"
  button**.

### Deferred (documented follow-ups — do NOT build here)
- SQS `eligibility-worker` polling `dental-eligibility-queue`.
- EventBridge 3-day pre-appointment batch (`pre_appointment_batch` trigger).
- Appointment-slot eligibility badge.
- Verification-queue page (today's patients + status).
- DentalXChange adapter (Stedi is prod primary per research_17).
- MassHealth (CKMA1) hardcoded frequency-limit reference data /
  `payer_id_overrides`.
- Secondary insurance COB auto-calculation.

### Decisions locked during brainstorming
- **Build slice:** sync-first; defer the batch worker.
- **Money units:** integer cents for all dollar fields (deductible, OOP,
  annual max), matching the cross-module "absolute cents everywhere" rule.
  Coinsurance stays a decimal fraction — the one documented exception.
- **Clearinghouse:** Stedi for dev *and* prod (research_17 supersedes
  research_13's DentalXChange-for-prod note; DentalXChange pricing never
  verified). Build only the Stedi adapter; keep it behind the interface.
- **Stedi verification approach:** build against the documented contract, unit-
  test against recorded 271-response fixtures, mock HTTP. The real sandbox
  smoke test is deferred to Staging Checkpoint 5 (first outbound call from the
  ECS private subnet — NAT/SSM/IAM untestable locally).
- **Parse depth:** robust common-case + null-safe (all fields Module 6 needs);
  defer MassHealth refdata.
- **Frontend:** chart benefit-summary card + verify button only.

## 2. Provider abstraction

Location: `apps/api/app/services/eligibility/`

### `base.py`
- `EligibilityProvider` ABC with two methods:
  - `async def check_eligibility(self, request: EligibilityRequest) -> EligibilityResult`
  - `async def search_payer(self, query: str) -> list[dict]`
- Dataclasses `EligibilityRequest`, `EligibilityResult`, `CategoryBenefit`,
  enums `EligibilityStatus` / `BenefitCategory`. **Adapted from research_13's
  interface, except all dollar fields are `int` (cents), not `Decimal`.**
- `EligibilityProviderError(message: str, retryable: bool = False)` — the
  adapter raises this; the router maps it to `failed` (retryable/transport) or
  `not_supported` (payer-not-found / AAA rejection).

### `stedi.py` — `StediProvider`
- Builds the Stedi v3 request:
  - Endpoint `POST https://healthcare.us.stedi.com/2024-04-01/change/medicalnetwork/eligibility/v3`
  - `Authorization: ApiKey <key>` header.
  - `tradingPartnerServiceId` = the payer's Stedi ID.
  - `serviceTypeCodes` always `["35","27","F3","AJ"]` (Dental Care / Specialty
    / Accident / Preventive).
  - `controlNumber` unique, ≤ 9 digits.
  - `dateOfBirth` `YYYYMMDD`, `dateOfService` `YYYYMMDD`.
- Calls via `httpx` with a **30s timeout**; transport/5xx/timeout →
  `EligibilityProviderError(retryable=True)`; payer-not-found / AAA →
  raise non-retryable (router → `not_supported`).
- Parses the `benefitsInformation` array → `EligibilityResult` (see §3).

### Credentials (runtime, from SSM — never inline)
- Stedi API key: read from the practice's `clearinghouse_api_key_ssm_path` via
  SSM Parameter Store at request time.
- `clearinghouse_submitter_id` from the practice record; provider NPI from the
  practice/provider record.
- These `practices` columns already exist (`clearinghouse_provider`,
  `clearinghouse_api_key_ssm_path`, `clearinghouse_submitter_id`).

## 3. Parsing (the core of 5.3)

Iterate `benefitsInformation` keyed by `code` × `serviceTypeCodes` ×
`coverageLevelCode` × `timeQualifierCode`:

| `code` | Meaning            | `coverageLevelCode` | `timeQualifierCode` |
|--------|--------------------|---------------------|---------------------|
| `1`    | Active coverage    | `IND` Individual    | `23` Calendar-year total |
| `6`    | Inactive           | `FAM` Family        | `29` Remaining      |
| `A`    | Co-insurance       |                     | `26` Lifetime       |
| `C`    | Deductible         |                     |                     |
| `F`    | Limitations (annual max / frequency) |   |                     |
| `G`    | Out-of-pocket max  |                     |                     |

Rules:
- **Status** from the first `code=1`/`6` in the subscriber loop; ignore the
  dependent loop (avoids contradictory active/inactive).
- **Deductible** (`C`): individual + family totals and met amounts.
- **Annual max** (`F`): qualifier `23` = total, `29` = remaining; correlate all
  `F` segments.
- **OOP max** (`G`).
- **Coinsurance** (`A`): **NOT summarized in this slice.** The original plan
  bucketed coinsurance into preventive/basic/major/ortho by scanning description
  keywords. A live Stedi sandbox call (Cigna, 2026-06-12) proved real dental
  271s return coinsurance **per individual CDT procedure code**, grouped under
  granular dental service-type codes (23/24/25/26/28/36/38/39/40/41/4/7), with
  the description holding *lists of D-codes* — no preventive/basic/major/ortho
  keywords. The 4-category model is lossy and misleading against real data, so
  the keyword categorizer was removed; the `coinsurance_*` summary fields stay
  `None`. The full per-code rates are preserved in `raw_response`. A faithful
  per-CDT-code coinsurance model is a **Module 6 (co-pay) design decision**
  (see §9) — Module 6's accuracy depends on getting it right.
- **Ortho** has no dedicated service-type code — scan the plan description
  case-insensitively for "ORTHO".
- **Never default a missing field to zero — store `null`.** Many payers omit
  remaining-deductible; null ≠ zero.

## 4. Data model — `eligibility_checks`

SQLAlchemy model at `apps/api/app/models/eligibility_check.py`, `TimestampMixin`
(soft delete), Alembic migration in `apps/api/alembic/versions/`. Mirrors
`packages/shared-types/src/schemas/eligibility.ts` (already wired into
`pnpm generate`), with two server-side additions not exposed in the API schema:
`raw_response` and the soft-delete column.

Columns:
- Identity / FKs: `id`, `practice_id`, `patient_id`, `patient_insurance_id`,
  `appointment_id` (nullable).
- `idempotency_key TEXT NOT NULL UNIQUE` — for manual checks this is **the
  request's `Idempotency-Key` header value** (naturally dedups client retries;
  the future batch will use `{appointment_id}_3day_precheck_{date}`).
- `status` ∈ `pending | verified | failed | not_supported` (default `pending`).
- `trigger` ∈ `manual | pre_appointment_batch` (only `manual` used in this
  slice).
- `clearinghouse` (`stedi` for now; `manual`/`dentalxchange` reserved),
  `payer_id_used`, `payer_name`, `plan_name`, `failure_reason`.
- Coverage: `coverage_status` ∈ `active | inactive | unknown`,
  `coverage_start_date`, `coverage_end_date`.
- **Dollar fields as integer cents** (nullable): `deductible_individual`,
  `deductible_individual_met`, `deductible_family`, `deductible_family_met`,
  `oop_max_individual`, `oop_max_individual_met`, `annual_max_individual`,
  `annual_max_individual_used`, `annual_max_individual_remaining`.
- **Coinsurance as `NUMERIC(5,4)` fraction** (patient share, nullable):
  `coinsurance_preventive`, `coinsurance_basic`, `coinsurance_major`,
  `coinsurance_ortho`.
- Waiting periods (integer months, nullable): `waiting_period_basic_months`,
  `waiting_period_major_months`, `waiting_period_ortho_months`.
- `frequency_limits` JSONB (nullable).
- `raw_response` JSONB **nullable** (null until a response arrives; **not
  serialized in API responses** — server-internal for debugging / manual
  review).
- Timestamps: `requested_at`, `verified_at`, `failed_at`, `created_at`,
  `updated_at`, `deleted_at`.

Indexes: `(practice_id, patient_id)`, `(patient_insurance_id)`, unique
`idempotency_key`, partial on `status WHERE status = 'pending'`.

### Schema-generation note
Two edits to `packages/shared-types/src/schemas/eligibility.ts`, then re-run
`pnpm --filter @dental/shared-types generate` so `generated.py` matches (do
**not** hand-edit `generated.py`):
1. Change all dollar fields (`deductible*`, `oopMax*`, `annualMax*`) from
   `z.number()` to `z.number().int()` (cents).
2. Add `planName: z.string().optional()` — the chart card displays it and the
   parser fills it; it is currently missing from the schema (only `payerName`
   exists).

With (2) done, the only columns on the model *not* in the API schema are the
server-internal `raw_response` and the soft-delete `deleted_at`.

## 5. API — `apps/api/app/routers/eligibility.py`

Every endpoint: practice-scoped (`_require_practice_scope`), **`eligibility_
verification` feature-flag gate** (`practice.features.get(...)`; 403 if off),
auto audit + idempotency middleware. The mutation also requires write-role
(`_require_write_role`).

### `POST /api/v1/eligibility/check`
- Body: `CreateEligibilityCheck` = `{ patientInsuranceId, appointmentId? }`.
- Flow:
  1. Resolve the `patient_insurance` row (must belong to the practice).
  2. Insert an `eligibility_checks` row `status='pending'`,
     `trigger='manual'`, `idempotency_key=<header>`, `requested_at=now`.
  3. Build `EligibilityRequest`, call `StediProvider.check_eligibility`
     inline (30s).
  4. On success → parse → update row to `verified`, fill structured fields +
     `raw_response`, set `verified_at`.
  5. On `EligibilityProviderError(retryable=True)` or timeout → `failed` +
     `failure_reason`, `failed_at`. On payer-not-found / AAA → `not_supported`
     + `failure_reason` (+ payer portal hint). **Never silently skip.**
  6. Return the `EligibilityCheck`.
- Secondary insurance may be checked (store the result) but **no COB
  calculation** — flag for manual review only.

### `GET /api/v1/eligibility/{checkId}`
- Fetch one check (practice-scoped). 404 if not found.

### `GET /api/v1/eligibility?patientId=<uuid>`
- Returns the **latest check per `patient_insurance_id`** for that patient —
  the shape the chart card consumes (one current benefit summary per insurance
  on file).

## 6. Frontend — chart card + verify button

- **Benefit-summary card** on the patient chart, fed by
  `GET /eligibility?patientId=`:
  - Header: coverage status + `verified as of <timestamp>`.
  - Plan name, deductible (individual) and annual-max **remaining shown as
    estimates** — `~$20 remaining*` — with the staleness disclaimer
    (`* As of last payer update. May not reflect claims processed in the last
    1–7 days.`).
  - Coinsurance % by category (preventive / basic / major / ortho).
  - Renders `failed`, `not_supported`, and `not-checked` states distinctly.
    Null fields render `—`, **never `$0`**.
- **"Verify now" button** → `POST /eligibility/check` for the selected
  `patient_insurance` row; shows in-flight state, then refreshes the card with
  the result. Available per insurance row (primary or secondary).

## 7. Testing

- **Parser/adapter unit tests** against recorded 271-response fixtures (HTTP
  mocked):
  - active + full benefits, inactive, payer-not-found / AAA rejection,
    missing-deductible (null preserved, not zero), coinsurance in both
    conventions (→ normalized to patient share), ortho-in-description.
- **Router tests** (established auth-patch pattern, `test_providers.py` style):
  - feature flag off → 403, happy path → `verified` row, Stedi transport error
    → `failed` row, payer-not-found → `not_supported`, idempotent retry returns
    the cached response.
- **Live Stedi sandbox smoke test → Staging Checkpoint 5** (cannot run from
  local dev; needs the ECS private subnet + SSM + IAM).

Per repo convention: every new exported function / endpoint ships with a test
(happy path + a failure case) in the same change.

## 8. Error handling summary

| Condition | Status | Notes |
|-----------|--------|-------|
| Stedi success | `verified` | structured fields + `raw_response` stored |
| Transport error / 5xx / 30s timeout | `failed` | `failure_reason`; staff can re-verify |
| Payer not found / AAA rejection | `not_supported` | first-class status + portal hint; never silently skip |
| Feature flag off | — | 403 before any work |
| Insurance not in practice | — | 404 |

## 9. Deferred follow-ups (next specs)

1. **Async batch pipeline:** EventBridge cron (3 days out) → SQS
   `dental-eligibility-queue` → ECS `eligibility-worker` (idempotency key
   `{appointment_id}_3day_precheck_{date}`); same provider + parser reused.
   This is what Staging Checkpoint 5 ultimately verifies end-to-end.
2. **Appointment-slot badge** (verified / pending / failed / not-checked).
3. **Verification-queue page** (today's patients + status as a work-list).
4. **DentalXChange adapter** behind the same interface (if/when prod needs a
   dental-specific fallback).
5. **MassHealth (CKMA1)** hardcoded frequency-limit reference data +
   `payer_id_overrides` lookup.
6. **Secondary insurance COB** (Module 6 territory).
7. **Per-CDT-code coinsurance model (Module 6 foundation).** Real dental 271s
   return coinsurance per procedure code under granular STCs (confirmed via live
   Stedi sandbox, 2026-06-12), not by 4 categories. Design how Module 5 should
   structure this (likely a CDT-code → patient-share map sourced from the 271)
   so Module 6 can look up the specific procedure's coinsurance. Until then the
   `coinsurance_*` summary fields are `None` and the per-code data lives only in
   `raw_response`. The chart card's per-category coinsurance rows render `—` in
   the meantime (rework with this model).
