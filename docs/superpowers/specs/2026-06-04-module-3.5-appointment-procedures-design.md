# Module 3.5 — Per-Appointment Procedures (Design)

**Status:** Approved design, ready for implementation plan
**Phase:** 3 (Billing & Insurance Depth) — first sub-project; see `phase3-build-order.md`
**Source plan:** `phase1_build_plan.md` § "Module 3.5 — Per-Appointment Procedures (Lite)"

## Purpose

Let the dentist record the procedure(s) actually performed at a visit — CDT code,
name, tooth/surface, billed fee, and an optional manually-entered insurance/patient
split. This is a scoped-down slice of Module 6 (the full co-pay engine): **capture the
data now; do not build the calculation service**. When Module 6 lands it consumes these
rows verbatim, with no data migration.

Procedures recorded here are what *happened* at the appointment. They are distinct from
treatment-plan items (what the dentist *intends* to do, Phase 2). The two stay
independent for now; a future "complete this plan item → create the procedure" link is
deliberately left as a nullable FK to add later.

## Non-goals (explicitly deferred)

- **No auto-calculation** of patient responsibility from eligibility data — Module 6.
- **No treatment-plan linking** — future; add a nullable FK then.
- **No full CDT catalog** — seed ~20 common codes; the complete D0100–D9999 catalog is Module 6.2.
- **No claims generation** — Module 7 reads these rows; not in scope here.
- **Not feature-flagged** — this is core scheduling data, available to every practice
  (unlike Modules 5/6/7 which are opt-in).

## Data model

Two new tables. A single migration `0025_appointment_procedures.py` creates both
(0024 is the latest existing revision; confirm at implementation time in case others
merge first).

### `cdt_codes` — reference catalog (seed data, no PHI)

Uses `TimestampMixin` (not `PHIMixin`) — reference data, no patient information.

| column | type | notes |
|--------|------|-------|
| `id` | UUID PK | |
| `code` | TEXT NOT NULL UNIQUE | e.g. `D1110` |
| `description` | TEXT NOT NULL | e.g. `Prophylaxis - adult` |
| `category` | TEXT NOT NULL | CHECK in (`diagnostic`,`preventive`,`basic`,`major`,`ortho`,`other`) |
| `default_fee_cents` | INTEGER NULL | practice-level fee override comes in Module 6 |
| `is_active` | BOOLEAN NOT NULL DEFAULT TRUE | |

Index: `UNIQUE (code)` only — small table, typeahead search uses a prefix/ILIKE scan.

**Seed (~20 common codes)**, applied in the migration:
D0120, D0140, D0150, D0210, D0220, D0274, D1110, D1120, D1206, D1208,
D2140, D2150, D2330, D2391, D2392, D2740, D2750, D3220, D4341, D4342.
Categories assigned per the ADA range table in `research/14_module6_copay_calculation.md` §3.
Descriptions from Open Dental's GPL seed data / ADA ranges (authoritative CDT book purchase is a Module 6 task).

### `appointment_procedures` — per-visit rows (PHI)

Uses `PHIMixin` (provides `created_at`, `updated_at`, `deleted_at`, `last_accessed_by`,
`last_accessed_at`) like `treatment_plan_items`. Fields mirror `treatment_plan_items`
where possible so the two tables stay shape-compatible for future linking.

| column | type | notes |
|--------|------|-------|
| `id` | UUID PK | |
| `practice_id` | UUID NOT NULL | tenant scope |
| `appointment_id` | UUID NOT NULL | FK → appointments, ON DELETE CASCADE |
| `patient_id` | UUID NOT NULL | denormalized from appointment for chart-level queries + audit |
| `cdt_code_id` | UUID NULL | FK → cdt_codes; nullable = free-text entry |
| `procedure_code` | TEXT NULL | denormalized `D1110`; **required if `cdt_code_id` is null** |
| `procedure_name` | TEXT NOT NULL | |
| `tooth_number` | TEXT NULL | single tooth or range |
| `surface` | TEXT NULL | e.g. `MOD` |
| `fee_cents` | INTEGER NOT NULL | billed fee |
| `insurance_est_cents` | INTEGER NULL | manual estimate; NULL = unknown |
| `patient_est_cents` | INTEGER NULL | manual estimate; NULL = unknown |
| `estimate_source` | TEXT NULL | CHECK in (`manual`,`eligibility`,`prior_eob`); provenance of the estimate |
| `notes` | TEXT NULL | |

**Coverage representation decision:** absolute cents (`insurance_est_cents` /
`patient_est_cents`), matching `treatment_plan_items`, the Module 6 engine output,
and the claims/ERA pipeline. A coverage *percentage* is **not** stored here — the only
place a percentage lives is the Module 5 eligibility coinsurance rate, which Module 6
consumes as an input and converts to cents. `estimate_source` records where a manual
estimate came from. See `phase3-build-order.md` for the cross-module rationale.

Constraints / indexes:
- CHECK: `cdt_code_id IS NOT NULL OR procedure_code IS NOT NULL` (one of the two present)
- CHECK: `estimate_source IN ('manual','eligibility','prior_eob')` (nullable)
- Index `(appointment_id)` — list procedures for an appointment
- Index `(patient_id, created_at)` — procedure history tab, newest first
- Index `(cdt_code_id)`
- No FK to `patient_insurance` yet — coverage is captured standalone; wiring to an
  eligibility-verified plan is Module 5/6.

## API

New router `apps/api/app/routers/appointment_procedures.py`, registered in the app
factory. Patient/appointment-scoped, following the existing `treatment_plans.py`
pattern (`_require_practice_scope`, `_require_write_role`, `_phi_audit_*` helper).

| method | path | notes |
|--------|------|-------|
| GET | `/api/v1/appointments/{id}/procedures` | list (active rows; soft-deleted excluded) |
| POST | `/api/v1/appointments/{id}/procedures` | add; `Idempotency-Key` required |
| PATCH | `/api/v1/appointments/{id}/procedures/{procId}` | edit |
| DELETE | `/api/v1/appointments/{id}/procedures/{procId}` | soft delete (`deleted_at`) |
| GET | `/api/v1/cdt-codes?q=...` | typeahead over seed list (code prefix + description ILIKE), active only |

Behavior:
- Every mutation writes a PHI audit entry tied to the patient (touch `last_accessed_by`/
  `last_accessed_at` on the procedure row, matching `_phi_audit_plan`).
- Idempotency enforced by the existing middleware (all mutation endpoints already require
  `Idempotency-Key`).
- Authorization: a user scoped to Practice A cannot read or mutate procedures on an
  appointment owned by Practice B → 404.
- Coverage/fee validation at the API boundary: `fee_cents >= 0`,
  `insurance_est_cents`/`patient_est_cents` >= 0 when present, `estimate_source` in the
  allowed enum when present.

### Totals

The list endpoint returns a `totals` object computed server-side, never on the client:
`{ feeCentsTotal, insuranceEstCentsTotal, patientEstCentsTotal }` — simple SUM over
active rows (NULL estimates treated as 0 in the sum but surfaced as unknown per-row).

## Shared types / schemas

Schemas are authored as Zod in `packages/shared-types/src/schemas/scheduling.ts` (or a
new `procedures.ts`) and the Pydantic models are **generated** via
`pnpm --filter @dental/shared-types generate` into `apps/api/app/schemas/generated.py`.
Do not hand-write Pydantic. New schemas:
`CdtCode`, `AppointmentProcedure`, `CreateAppointmentProcedure`,
`UpdateAppointmentProcedure`, `AppointmentProcedureListResponse` (rows + totals).

## Frontend

Two surfaces, both consuming the typed API client (`apps/web/lib/api/...`).

1. **Procedures section in the appointment modal** (`apps/web/...AppointmentModal`):
   a table of procedure rows with add/edit/delete. Row editor: CDT typeahead (falls back
   to free-text name + code), tooth/surface, fee, optional insurance/patient estimate
   with source dropdown. Bottom totals strip: `Fee total · Insurance est. · Patient est.`
2. **Procedure History tab on the patient chart**: read-only list of all procedures
   across appointments, sorted by appointment `start_time` desc.

## Testing

Per CLAUDE.md: new endpoints ship with tests in the same change (happy path + a failure
case). All non-integration tests run before claiming done.

- **Unit:** add/edit/delete/list serialization; totals computation (including NULL
  estimates); validation bounds (negative fee rejected, bad `estimate_source` rejected);
  free-text vs CDT-code constraint (`cdt_code_id` null requires `procedure_code`).
- **Integration:** create appointment → add 3 procedures → totals match; delete one →
  totals recompute; deleting the appointment soft-deletes its procedures (cascade via
  service layer).
- **Authorization:** user scoped to Practice A cannot add/read a procedure on Practice B's
  appointment (404).
- **Audit:** a mutation writes the audit fields on the row.
